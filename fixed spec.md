## Видение

Проект должен перестать доверять клиенту в вопросах шахматной логики и прав на действие. После фикса сервер должен быть единственным источником истины для легальности хода, принадлежности игрока к партии, статуса матча и истории ходов, а клиент должен выполнять только UX-функцию. Это уменьшит риск порчи партии, прямых API-эксплойтов и скрытых рассинхронов после `undo`, перезагрузки и игры по кастомным правилам.

**Ключевые требования:**
- Нелегальный ход через прямой API-запрос не применяется.
- Игровые действия разрешены только участникам соответствующей партии.
- История ходов и производные поля формируются только на сервере.
- Logout завершает реальную серверную сессию.
- Ошибки сервера не приводят к падению процесса.

## Контекст и бизнес-ценность

### Проблема
- Сервер недостаточно валидирует ходы и частично доверяет данным клиента.
- Игровые endpoints не привязаны строго к сессии пользователя.
- Logout не завершает серверную авторизацию.
- Некоторые ошибки обрабатываются небезопасно и могут ломать runtime.

### Кому это важно
- Игрокам: партия должна быть честной и воспроизводимой.
- Владельцу проекта: мультиплеер не должен позволять очевидные exploit-сценарии.
- Разработке: логика игры должна быть тестируемой и детерминированной.

### Бизнес-результат
- Снижение числа логических багов и “сломанных” партий.
- Переход от demo-поведения к рабочей multiplayer-модели.
- Повышение доверия к custom rules и online-режиму.

## Scope

### Входит в задачу
- Переработка публичных request schemas.
- Server-side валидация хода для всех поддерживаемых правил.
- Привязка игровых действий к `session.userId`.
- Cleanup небезопасных game endpoints.
- Исправление logout.
- Улучшение регистрации и обработки duplicate constraints.
- Добавление automated tests и test script.

### Не входит в задачу
- Редизайн UI авторизации и игрового экрана.
- Полная миграция старых партий на новую модель совместимости.
- Новые игровые режимы сверх уже существующих.

- Нельзя ломать текущие сценарии: создание игры, присоединение по коду, ход, promotion, draw, `void`.

## Функциональные требования

### FR-1. Публичные API-схемы отделены от DB insert-схем
- Нельзя использовать insert-схемы таблиц как схемы внешнего API.
- Для каждого mutating endpoint должны быть отдельные request schemas.
- Для endpoint хода клиент передает только входные параметры действия, а не вычисленные сервером поля.

### FR-2. Сервер полностью валидирует легальность хода
- Сервер проверяет геометрию хода фигуры.
- Сервер проверяет блокировки по пути.
- Сервер проверяет взятия, en passant, castling, promotion и special rules.
- Сервер проверяет, что после применения хода позиция остается допустимой по king safety.
- Запрос с нелегальным ходом всегда получает `400`.

### FR-3. Сервер является источником истины для истории партии
- Поля `player`, `piece`, `captured`, `special`, `moveNumber` вычисляются на сервере.
- История ходов хранится в нормализованном виде.
- `rebuildGameStateFromMoves` восстанавливает состояние без зависимости от клиентских допущений.

### FR-4. Все игровые действия авторизованы по сессии
- Сервер должен использовать `session.userId` для определения текущего пользователя.
- Создание игры создает партию от текущего пользователя.
- Join использует текущего пользователя, а не случайный `playerId`.
- Игрок, не являющийся участником партии, не может ходить, откатывать, предлагать ничью, принимать ничью или менять статус партии.

### FR-5. Опасные state endpoints ограничены
- Произвольная смена `status` клиентом запрещена.
- Обновление `capturedPieces` клиентом запрещено либо переводится во внутренний механизм.
- Смена состояния партии допускается только через проверенные сценарии:
  - resign
  - draw accept/decline
  - auto-complete по checkmate/stalemate

### FR-6. Logout завершает серверную и клиентскую сессию
- Frontend вызывает `POST /api/auth/logout`.
- После logout очищаются `localStorage`, runtime state и React Query cache.
- После reload пользователь не должен автоматически восстанавливаться в authenticated state.

### FR-7. Регистрация корректно обрабатывает duplicate phone
- До вставки пользователя проверяются `username`, `email`, `phone`.
- Duplicate constraint по телефону возвращает понятную business error с кодом `400`.

### FR-8. Global error handling безопасен
- Error handler не должен повторно бросать ошибку после отправки ответа.
- Ошибка логируется, но runtime продолжает работать.

---

## Нефункциональные требования

### NFR-1. Надежность
- После `undo`, refresh и повторной загрузки состояние партии не должно расходиться с историей.

### NFR-2. Безопасность
- Запросы извне не должны позволять ход за чужой цвет или подмену служебных полей.

### NFR-3. Тестопригодность
- Основная бизнес-логика должна быть вынесена в тестируемые модули.
- В проекте должен появиться `test` script.

### NFR-4. Совместимость
- UI может остаться близким к текущему, но не должен зависеть от `localStorage playerId` как от источника прав на действие.

## Acceptance Criteria

### AC-1. Безопасность игровых действий
- Пользователь вне партии получает `401` или `403` на все mutating game endpoints.
- Пользователь не может сделать ход за чужой цвет даже при ручной подмене request body.

### AC-2. Легальность хода
- Нелегальный ход через API возвращает `400`.
- Сервер не допускает bypass клиентской логики через прямой fetch/Postman/curl.

### AC-3. Согласованность состояния
- `undo` не ломает ход партии.
- Перезагрузка игры не приводит к расхождению board state и move history.

### AC-4. Auth и session behavior
- Logout завершает сессию на сервере и на клиенте.
- После refresh пользователь не оказывается “залогинен обратно” без новой сессии.

### AC-5. Регистрация
- Duplicate `username`, `email`, `phone` возвращают ожидаемые бизнес-ошибки, а не общий `500`.

### AC-6. Runtime stability
- Исключение в route handler не роняет процесс.

## Test Scenarios

### API Authorization
- Гость без сессии пытается сделать ход.
- Пользователь A пытается сделать ход в партии пользователя B.
- Пользователь A пытается вызвать `undo` в чужой партии.
- Пользователь A пытается принять ничью в чужой партии.

### Move Validation
- Ладья пытается пройти через фигуру.
- Слон пытается сходить не по диагонали.
- Пешка делает недопустимый ход.
- Игрок пытается оставить своего короля под шахом.
- Проверка `double-knight` на live route и после `undo`.

### Session & Logout
- Пользователь логинится, выходит, перезагружает страницу.
- Гостевой вход создает рабочую сессию, а не только локальный клиентский state.

### Regression Smoke
- Создание обычной партии.
- Join по share code.
- Игра в `fischer-random`.
- Базовый сценарий `void` с ходом на двух досках.

## Риски

- Логика правил сейчас частично размазана между client и server; при фиксе есть риск временных расхождений.
- `void` и `double-knight` имеют наибольшую вероятность регрессий, потому что влияют на turn model.
- Перевод на session-based ownership может затронуть старые партии и текущий flow guest mode.

---

## Решения вне scope

- Миграция legacy game records на новую совместимую ownership-модель.
- Полный продуктовый redesign auth/game UX.
- Добавление новых custom rules.
