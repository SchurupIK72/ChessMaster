## Видение

В ChessMaster должен появиться полноценный шахматный таймер с выбором контроля времени до начала партии. Таймер должен быть authoritative на сервере, корректно переживать refresh и reconnect, автоматически переключаться между игроками после завершения хода и завершать партию по timeout. Для режима `Void` должен использоваться один общий таймер на сторону для обеих досок, а переключение должно происходить только после завершения полного хода игрока по правилам `Void`, а не после первого под-хода.

**Ключевые требования:**
- Перед созданием партии пользователь выбирает длительность таймера в отдельном modal flow.
- Выбранный time control сохраняется в матче и одинаково отображается всем участникам и зрителям.
- Отсчёт времени ведётся на сервере, а клиент только отображает актуальное состояние.
- После успешного завершения хода одного игрока его таймер останавливается, а таймер соперника запускается.
- В `Void` у каждой стороны только один таймер на обе доски.
- Timeout должен завершать игру предсказуемо и без возможности сделать дополнительный ход после истечения времени.

## Контекст и бизнес-ценность

### Проблема
- Сейчас в проекте нет chess clock, поэтому партия не ограничена по времени и не поддерживает стандартный режим игры с time pressure.
- В текущем pre-game flow нет выбора time control, поэтому матч нельзя запускать в формате bullet, blitz или rapid.
- Клиент и сервер не хранят authoritative clock state, из-за чего после refresh, reconnect и `undo` невозможно корректно восстановить оставшееся время.
- В режиме `Void` ход состоит из нескольких действий, поэтому наивное переключение таймера после каждого POST на ход приведёт к неправильному игровому поведению.

### Кому это важно
- Игрокам: чтобы играть полноценные партии с ограничением времени.
- Зрителям: чтобы видеть актуальный countdown и понимать темп партии.
- Владельцу проекта: чтобы продукт поддерживал ожидаемую шахматную механику и не расходился между клиентом и сервером.

### Бизнес-результат
- Появляется полноценный time-control gameplay loop.
- Улучшается соревновательный характер матчей.
- У проекта появляется база для будущих режимов вроде increment/delay без переписывания базовой архитектуры.

## Scope

### Входит в задачу
- Отдельное modal window выбора time control до создания партии.
- Сохранение выбранного контроля времени в матче.
- Server-authoritative countdown для `white` и `black`.
- Переключение таймеров после завершения хода в стандартном режиме.
- Корректное переключение таймеров после завершения полного хода в `Void`.
- Отображение таймера для игроков и spectator.
- Автоматическое завершение партии по timeout.
- Корректное поведение после refresh, reconnect, SSE resync и `undo`.

### Не входит в задачу
- Increment, Bronstein delay, move delay или custom arbitrary time input.
- Отдельная кнопка pause/resume посреди активной партии.
- Разные таймеры для каждой доски в `Void`.
- FIDE-специфичные исключения по insufficient mating material на timeout.

## Ограничения и допущения

- MVP использует только базовый countdown без increment/delay.
- Допущение для MVP: пользователю предлагается фиксированный список preset-значений, например `1`, `3`, `5`, `10`, `15` минут; точный список можно скорректировать перед реализацией без изменения общей архитектуры.
- Таймер должен быть authoritative на сервере; клиент не может присылать произвольное оставшееся время.
- До момента, пока второй игрок не занял слот и матч не стал `active`, таймеры должны быть paused.
- При статусах `completed`, `draw`, `resigned` и `timeout` таймеры должны быть остановлены.
- В `Void` время стороны общее для обеих досок и продолжает идти, пока сторона не завершила весь свой ход по правилам режима.
- Наличие существующего `undo` означает, что clock state должен быть либо восстанавливаемым из истории, либо храниться в snapshot-совместимом виде.

## Пользовательские сценарии

### Сценарий 1. Создание партии с таймером
- Пользователь открывает flow создания новой партии.
- После выбора правил открывается modal выбора длительности таймера.
- Пользователь выбирает preset time control и подтверждает создание матча.
- Новый матч сохраняется с выбранным контролем времени, но countdown ещё не идёт, пока не подключится второй игрок.

### Сценарий 2. Старт активной партии
- Второй игрок присоединяется к матчу.
- Матч переходит в `active`.
- Запускается таймер стороны, которая должна делать первый ход, то есть `white`.

### Сценарий 3. Обычный ход в стандартной партии
- Активный игрок думает, и его время убывает.
- Сервер принимает валидный ход.
- Сервер фиксирует остаток времени сделавшего ход игрока, останавливает его таймер и запускает таймер соперника.
- Все клиенты получают обновлённое timer state через существующий live-update flow.

### Сценарий 4. Полный ход в Void
- Игрок делает первый под-ход на одной из двух досок.
- Его общий таймер продолжает идти.
- Игрок завершает обязательный второй под-ход, либо завершает ход через допустимый `void-transfer`, либо сервер auto-finalize-ит ход, если другая доска не даёт легального продолжения.
- Только после этого таймер активной стороны останавливается и запускается таймер соперника.

### Сценарий 5. Refresh или reconnect
- Игрок или spectator обновляет страницу в середине партии.
- Клиент получает с сервера текущее authoritative timer state.
- Countdown продолжается с корректного значения без визуального reset к начальному времени.

### Сценарий 6. Timeout
- У активной стороны заканчивается время.
- Сервер помечает матч как завершённый по timeout и назначает победителем соперника.
- После этого никакие новые ходы или предложения ничьей не должны менять состояние партии.

## Функциональные требования

### FR-1. Выбор time control до старта партии
- Перед созданием новой партии пользователь должен пройти отдельный шаг выбора длительности таймера.
- Этот шаг должен быть реализован как modal window или отдельный modal step внутри существующего pre-game flow.
- Пользователь должен выбрать ровно один preset time control.
- Создание партии без выбранного time control не допускается.
- Выбранное значение должно быть явно показано пользователю до подтверждения создания матча.

### FR-2. Сохранение time control в матче
- Матч должен хранить выбранный базовый time control и текущее состояние таймера для обеих сторон.
- Состояние таймера должно включать как минимум:
- базовую длительность для стороны;
- оставшееся время `white`;
- оставшееся время `black`;
- признак, чей таймер сейчас активен, либо состояние `paused`;
- authoritative timestamp, от которого считается текущий активный отрезок времени.
- Клиентские запросы получения матча должны возвращать timer state вместе с остальным состоянием игры.

### FR-3. Lifecycle таймера
- Пока матч находится в состоянии ожидания второго игрока, оба таймера должны быть paused.
- Когда матч переходит в `active`, должен стартовать таймер стороны `white`.
- Когда матч завершается любым финальным статусом, таймеры должны быть остановлены.
- Повторное открытие матча не должно заново запускать или сбрасывать таймер.

### FR-4. Переключение таймера в стандартной партии
- В стандартной партии переключение таймера должно происходить только после успешного server-side принятия легального хода.
- Если ход отклонён сервером как невалидный, таймер не должен переключаться.
- Если ход требует дополнительного выбора фигуры для promotion, таймер должен переключаться только после окончательного подтверждения promotion move.
- После переключения текущий игрок больше не может выполнить ещё один ход до следующего своего turn.

### FR-5. Переключение таймера в Void
- Для `Void` у каждой стороны должен существовать только один общий таймер на обе доски.
- Первый под-ход на одной из досок не должен останавливать таймер активной стороны.
- Если `Double Knight` требует дополнительный hop в рамках того же полного хода, таймер активной стороны должен продолжать идти до полного завершения этой последовательности.
- Если ход завершается через `void-transfer`, переключение таймера должно происходить после server-side finalize full turn.
- Если сервер по правилам `Void` auto-finalize-ит ход, потому что на другой доске нет легального продолжения, переключение таймера должно происходить в этот же момент.
- До полного завершения хода в `Void` соперник не должен получать активный таймер.

### FR-6. Timeout и завершение партии
- Если оставшееся время активной стороны достигает `0`, матч должен автоматически завершаться.
- Победителем по timeout должен считаться соперник стороны, у которой истекло время.
- UI должен явно показывать, что партия завершилась именно по timeout, а не по checkmate или resign.
- После timeout сервер должен блокировать новые ходы, `undo`, draw-flow и другие mutating actions, если для них не определён отдельный административный сценарий.

### FR-7. Отображение таймера в UI
- Оба таймера должны быть видимы в игровой странице в течение всей партии.
- Активный таймер должен визуально отличаться от paused timer.
- Spectator должен видеть те же значения таймера, но без возможности управлять ими.
- Низкий остаток времени должен иметь заметную визуальную индикацию.
- Отображение должно быть корректным как для обычной игры, так и для `Void`.

### FR-8. Reconnect, refresh и live sync
- После refresh клиент должен восстанавливать countdown из server response, а не из локального предположения.
- При reconnect или повторном открытии canonical match URL пользователь должен видеть актуальный timer state.
- SSE или другой live-update механизм должен доставлять изменения clock state одновременно с изменением turn/state.
- Если клиент временно потерял соединение, после повторной синхронизации он должен получить актуальный remaining time без ручного вмешательства.

### FR-9. Undo и восстановление состояния
- Если `undo` остаётся доступным в продукте, clock state после `undo` должен восстанавливаться детерминированно.
- Повторная сборка состояния матча из начального состояния и истории ходов должна давать тот же timer state, что и живой матч.
- Для этого система должна сохранять достаточно данных о clock state на каждом принятом ходе или иным образом обеспечивать воспроизводимое восстановление.

### FR-10. Авторизация и целостность
- Сервер должен быть единственным источником истины для timer state.
- Клиент не должен иметь возможности через API установить себе больше времени или вручную переключить активную сторону.
- При race condition или повторной отправке одного и того же хода таймер должен быть переключён не более одного раза.
- Timer state не должен становиться отрицательным даже при пограничных запросах около нуля.

## Нефункциональные требования

### NFR-1. Точность
- Drift между отображением на клиенте и authoritative server state должен оставаться в пределах визуально приемлемой погрешности, целевой ориентир для UX: не более 1 секунды после очередной синхронизации.

### NFR-2. Надёжность
- Refresh страницы, reconnect по `/match{matchId}` и открытие матча spectator-ом не должны сбрасывать или искажать таймер.

### NFR-3. Атомарность
- Смена `currentTurn`, обновление timer state, фиксация хода и смена статуса матча при timeout должны происходить как единый согласованный server-side flow.

### NFR-4. Совместимость
- Новая функциональность не должна ломать существующие режимы `void`, `double-knight`, `fog-of-war`, `meteor-shower`, spectator mode и reconnect по match link.

### NFR-5. Тестопригодность
- Решение должно покрываться automated tests на старт таймера, handoff, timeout, reconnect и `Void`-специфику.

### NFR-6. UX и доступность
- Состояние таймера должно различаться не только цветом, но и текстом/иконографикой, чтобы пользователь мог понять, чей ход и чей clock активен.

## Граничные случаи и специальные правила

- Если второй игрок ещё не присоединился, время белых не должно уменьшаться.
- Если активный игрок отправил невалидный ход, матч и таймер должны остаться в том же состоянии.
- Если у игрока время на исходе, а вкладка была неактивна, после возврата клиент должен показать уже уменьшившееся server-side значение.
- Если игра завершается checkmate, stalemate, draw или resign до истечения времени, countdown должен сразу остановиться.
- Если spectator открывает матч посреди активной партии, он должен увидеть актуальные таймеры без права вмешательства.
- В `Void` таймер не должен переключаться после первого под-хода, если полный ход ещё не завершён.
- В `Void` таймер должен корректно вести себя при `void-transfer`, `Double Knight` на одной доске и auto-finalize при отсутствии хода на второй доске.
- При `undo` нельзя получать дублирующее списание времени или восстановление к неверному часу.

## API и модель данных

### Требования к server contract
- `POST /api/games` должен принимать выбранный time control как часть payload создания матча.
- `GET /api/games/:id`, `GET /api/games/match/:matchId` и `GET /api/games/share/:shareId` должны возвращать timer state в shape, пригодном для немедленного рендера countdown.
- Ответы live-update flow должны позволять клиенту синхронно обновлять `currentTurn`, статус матча и таймер.

### Требования к persistent model
- Persistent game model должна хранить базовый time control и актуальное clock state.
- Persistent model должна позволять понять:
- сколько времени осталось у каждой стороны;
- чей clock сейчас активен;
- с какого server timestamp идёт текущий активный отсчёт;
- по какой причине матч завершился, если это timeout.
- История ходов или связанный snapshot-механизм должны содержать достаточную информацию для корректного `undo` и rebuild.

## Acceptance Checklist

- Пользователь выбирает time control до создания матча.
- Новый матч сохраняет выбранный таймер и отдаёт его через API.
- Таймеры не стартуют, пока в игре нет второго игрока.
- После старта матча работает стандартное переключение `white -> black -> white`.
- При `Void` используется один таймер на сторону для обеих досок.
- В `Void` handoff выполняется только после завершения полного хода.
- Spectator видит таймеры в read-only режиме.
- Refresh и reconnect не сбрасывают countdown.
- Timeout завершает матч и назначает победителя.
- `Undo` и rebuild не приводят к рассинхронизации времени.

---

## Риски реализации

- Наивная client-side реализация приведёт к drift и возможным спорам о времени.
- `Void` и `Double Knight` создают сложные точки завершения хода, где легко ошибиться с handoff.
- Если не предусмотреть timer snapshot/rebuild strategy, существующий `undo` начнёт восстанавливать игру в неверное время.
- Если timeout не будет частью атомарного server-side flow, возможны гонки между последним ходом и истечением времени.
