## Видение

Каждый матч должен иметь постоянный публичный идентификатор и каноническую ссылку вида `https://chessmasterx.onrender.com/match{matchId}`. Игроки партии должны по этой ссылке в любой момент возвращаться в свой матч без потери роли, а любой другой пользователь, открывший ту же ссылку, должен попадать в матч как `spectator` с доступом только на просмотр.

**Ключевые требования:**
- Каждый матч получает уникальный публичный `matchId`.
- Матч можно открыть напрямую по каноническому URL без ручного поиска или join-кода.
- Игрок восстанавливается в матче как `white` или `black`.
- Не-игрок открывает тот же матч как `spectator`.
- Spectator не может выполнить ни одно игровое действие.

## Контекст и бизнес-ценность

### Проблема
- Текущий flow завязан на внутренний `gameId`, `shareId` и join-механику, но не дает удобный постоянный URL матча.
- Игроку неудобно повторно подключаться к уже существующей партии по одной стабильной ссылке.
- У проекта нет формального spectator mode по ссылке матча.
- Нет четкого разделения read-only доступа зрителя и прав игрока.

### Кому это важно
- Игрокам: чтобы вернуться в активный матч в любой момент.
- Зрителям: чтобы открыть матч по ссылке и наблюдать за ходом партии.
- Владельцу проекта: чтобы матч имел понятный публичный URL и безопасный режим наблюдения.

### Бизнес-результат
- Улучшение reconnect UX для матчей.
- Упрощение шаринга партий.
- Появление безопасного spectator mode без вмешательства в игру.

---

## Scope

### Входит в задачу
- Новый публичный `matchId` для матчей.
- Новый канонический URL матча `https://chessmasterx.onrender.com/match{matchId}`.
- Открытие матча напрямую по URL.
- Серверное определение роли пользователя в матче: `white`, `black`, `spectator`.
- Reconnect существующих игроков по ссылке.
- Read-only spectator mode для любых других пользователей.
- Обновление invite/share UX на новый URL.

### Не входит в задачу
- Чат, комментарии и реакции зрителей.
- Счетчик зрителей и online-presence.
- Роль модератора или арбитра.
- Полный отказ от legacy `shareId`, если он нужен как временная совместимость.

---

## Ограничения и допущения

- Роль пользователя должна определяться только на сервере через session и данные матча.
- Spectator mode не должен зависеть от `localStorage playerId`.
- Канонический URL матча должен работать и после refresh, и после прямого открытия в новой вкладке.
- Если пользователь не является игроком матча, открытие ссылки не должно автоматически занимать свободный слот игрока.
- Для матчей с `fog-of-war` spectator считается наблюдателем и видит полное состояние матча.
- Допущение: `matchId` является отдельным публичным идентификатором матча и не обязан совпадать с внутренним DB `games.id`.

## Функциональные требования

### FR-1. Публичный идентификатор матча
- Каждый матч должен получать уникальный публичный `matchId` при создании.
- `matchId` должен быть стабилен на протяжении всей жизни матча.
- `matchId` должен быть уникален на уровне persistent storage.
- Новый матч нельзя создать без валидного публичного идентификатора.

### FR-2. Канонический URL матча
- Для каждого матча должен существовать канонический URL вида `/match{matchId}`.
- Прямое открытие этого URL должно загружать матч без промежуточного ручного ввода join-кода.
- При несуществующем `matchId` пользователь должен получать корректный `404`-сценарий.

### FR-3. Reconnect игрока
- Если пользователь по session является `whitePlayerId` или `blackPlayerId` данного матча, он должен открывать матч в соответствующей роли.
- Игрок, вернувшийся по ссылке, должен сохранять право делать доступные ему игровые действия.
- Повторное открытие ссылки игроком не должно требовать повторного join-flow.

### FR-4. Spectator mode
- Если пользователь открывает ссылку матча и не является игроком этой партии, он должен получать роль `spectator`.
- `spectator` должен видеть:
- доску или доски матча;
- историю ходов;
- текущий статус, captured pieces, draw state и live updates.
- `spectator` не должен иметь права:
- делать ход;
- делать `undo`;
- предлагать, принимать или отклонять ничью;
- сдаваться;
- менять статус матча;
- занимать слот игрока без отдельного явного join-действия.

### FR-5. Поведение пользователя без активной session
- Пользователь без активной session должен иметь возможность открыть матч по ссылке как spectator.
- Система должна обеспечить для этого безопасный guest-compatible flow без автоматической выдачи player-прав.

### FR-6. Поведение при свободном слоте игрока
- Если партия еще не заполнена двумя игроками, пользователь по умолчанию все равно открывает матч как spectator.
- Переход из spectator в player должен быть только отдельным явным действием, если это разрешено продуктовым сценарием.
- Само открытие ссылки не должно автоматически превращать spectator в игрока.

### FR-7. Серверное определение роли матча
- Сервер должен возвращать effective role пользователя для конкретного матча.
- Роль должна вычисляться как одна из: `white`, `black`, `spectator`.
- Все mutating game endpoints должны использовать эту роль для авторизации действий.

### FR-8. Share и invite flow
- После создания матча UI должен показывать и копировать канонический матч-URL.
- Текст приглашения и share-modal должны использовать новый URL как основной способ входа в матч.
- Если legacy `shareId` временно остается в системе, он должен быть secondary/backward-compatible, а не primary UX.

### FR-9. Live updates для зрителей
- Spectator должен получать те же live updates матча, что и игроки.
- При reconnect или открытии прямой ссылки spectator должен видеть актуальное состояние матча без ручной синхронизации.

### FR-10. Fog-of-war и spectator visibility
- Spectator не считается игроком для целей fog-of-war masking.
- Spectator должен видеть полный board state и полную историю даже в матчах с `fog-of-war`.

## Нефункциональные требования

### NFR-1. Безопасность
- Read-only режим зрителя должен быть enforce-нут на сервере, а не только скрытием кнопок на клиенте.

### NFR-2. Надежность
- Reconnect по канонической ссылке должен работать после refresh и после повторного открытия страницы.

### NFR-3. Совместимость
- Новая фича не должна ломать существующие сценарии создания матча, SSE, custom rules и player reconnect.

### NFR-4. Тестопригодность
- Решение должно покрываться automated tests на role resolution, spectator restrictions и direct URL open.

### NFR-5. Производительность
- Открытие матча по каноническому URL не должно требовать лишних последовательных запросов, если роль и матч можно определить в одном связном flow.

## Acceptance Criteria

### AC-1. Уникальный match URL
- Каждый матч получает уникальный публичный `matchId`.
- Канонический URL матча имеет формат `/match{matchId}`.

### AC-2. Reconnect игрока
- Игрок матча по канонической ссылке попадает в свой матч в правильной роли.
- Игрок после refresh не теряет привязку к матчу.

### AC-3. Spectator mode
- Любой пользователь, не являющийся игроком данного матча, открывает матч как spectator.
- Spectator не может выполнить игровое действие даже через прямой API-запрос.

### AC-4. Visibility
- Spectator видит актуальное состояние матча, историю и live updates.
- Spectator видит полную доску в матчах с `fog-of-war`.

### AC-5. Ошибки
- Несуществующий `matchId` возвращает корректный `404`.
- Невалидные попытки spectator выполнить mutating actions возвращают `401/403`.

### AC-6. Share UX
- UI копирует и показывает новый матч-URL как основную ссылку на матч.

## Test Scenarios

### Routing
- Создать матч и открыть его по `/match{matchId}` в новой вкладке.
- Открыть несуществующий `/match{matchId}`.

### Role Resolution
- Белый игрок открывает матч по ссылке.
- Черный игрок открывает матч по ссылке.
- Третий пользователь открывает тот же матч по ссылке.
- Пользователь без активной session открывает матч по ссылке.

### Spectator Restrictions
- Spectator пытается сделать ход.
- Spectator пытается сделать `undo`.
- Spectator пытается предложить ничью.
- Spectator пытается сдаться или сменить статус матча.

### Visibility
- Spectator открывает матч с `fog-of-war`.
- Spectator получает live updates во время игры.
- Игрок переподключается по ссылке в активный матч.

---

## Риски

- В проекте уже есть `shareId`, поэтому новая match-link модель может конфликтовать с текущим join/share UX.
- Клиент до сих пор частично опирается на `localStorage playerId`, и spectator flow нужно будет аккуратно развести с legacy логикой.
- Нужно не сломать существующие `void`, `double-knight`, SSE и spectator visibility для кастомных режимов.
